Self-Updating Skill — Design Spec

2026-07-15 · netsuite-cli update refreshes the agent skill (SKILL.md) alongside the binary, and a new skill install command bootstraps or refreshes it on demand — so release-binary users (no repo clone) keep the skill in sync with the tool.

1. Problem

Today netsuite-cli update swaps only the binary (release assets are the three platform binary zips — no SKILL.md). The skill stays current only for developers whose ~/.claude/skills/netsuite-cli is a symlink into a git-pulled repo clone. A user who installs from a release binary never receives skill updates from update at all. Goal (Jon, 2026-07-15): the skill self-updates with the binary, without clobbering the symlinked repo case.

2. Content source — embed + re-exec

  • SKILL.md is compiled into the binary: pub const EMBEDDED_SKILL: &str = include_str!("../../skills/netsuite-cli/SKILL.md"); (from src/commands/skill.rs, resolves to the repo's single-file skill). The embedded copy therefore always matches the binary's exact source version.
  • The stale-in-memory problem: update runs in the OLD process. After self_update replaces the executable file on disk, the running process still holds the OLD embedded skill — so it cannot write the new one directly. Resolution: after a successful binary swap, update spawns the freshly-installed binary (std::env::current_exe() now points at it) as <exe> skill install, which runs NEW code with the NEW embedded skill. Its stdout JSON folds into the update result under a "skill" key.
  • Re-exec failure (non-zero exit, unparseable output, missing exe) is non-fatal: the update still returns {"updated":true,…} exit 0, with "skill":{"installed":false,"reason":"<error>"} and a one-line stderr note. The primary operation succeeded.
  • update --check is read-only and never touches the skill. New update --no-skill flag skips the re-exec entirely.

4. Target resolution & guards

resolve_skill_target(dir_override, config_base, …) — a pure function returning either a write path or a skip reason:

  • Base dir: $CLAUDE_CONFIG_DIR if set, else ~/.claude. Target file = <base>/skills/netsuite-cli/SKILL.md.
  • --dir <path> overrides the base entirely; the target is <path>/SKILL.md (or, if <path> ends in SKILL.md, that exact file), parents created as needed — explicit user intent, no base-existence check.

Guards, checked in order; every skip is non-fatal and reported via the installed:false + reason shape (plus a stderr line where noted):

ConditionOutcomereason
No --dir and base dir does not existSkip; stderr tip naming skill install --dir <path>no Claude config dir
Target file, or its parent dir, is a symlinkSkip (protects the repo-symlink dev case)symlink — skill tracks its source repo via git
Target exists and its content already equals EMBEDDED_SKILLSkip (idempotent, quiet re-runs)already current
OtherwiseCreate skills/netsuite-cli if missing, write the file— (installed:true)

Symlink detection uses symlink_metadata (does not follow the link) on both the file and the parent dir. Writing always overwrites a non-symlink regular file — "refresh" is the intent; hand-edited managed skills are not a supported case.

5. Testing

  • Unit (pure resolver): table-driven over (dir_override present?, base set/exists?, target is symlink?, existing content matches?) → expected decision (WritePath vs Skip(reason)). Uses tempfile dirs; no globals mutated except a scoped CLAUDE_CONFIG_DIR guard in a single env-owning test (process env is global — mirror the existing EnvVarGuard precedent).
  • Integration (tests/skill_commands.rs, filesystem only, no network): skill install --dir <tmp> writes SKILL.md with the embedded content; second run reports already current; a symlinked target is skipped with the symlink reason; a missing base with no --dir skips with the config-dir reason.
  • Embedded-content assertion: EMBEDDED_SKILL is non-empty and starts with the frontmatter ---\nname: netsuite-cli.
  • Re-exec glue in update.rs stays thin and is not unit-tested (needs a real binary swap + network), matching how the existing check_for_update/install_update network paths are already left to manual/live verification. Covered by the final live check.

6. Docs & the one caveat

  • README "Updating" section: update now also refreshes the skill; explains the symlink/no-config-dir skips and --no-skill; documents skill install for first-time placement and the --dir override.
  • SKILL.md / README note that the skill is self-managing via the binary.
  • Version-skew caveat, documented plainly: the first update onto a skill-aware release is performed by the older, skill-unaware binary, so it refreshes only the binary that hop. From the next version onward the refresh is automatic; a user can run skill install once to bootstrap immediately.

7. Out of scope

  • Multi-file skills (the skill is a single SKILL.md today; embedding one file suffices).
  • Bundling SKILL.md as a separate release asset or fetching it over the network (embedding removes the need).
  • Managing skills for any tool other than netsuite-cli.
  • Removing/uninstalling the skill.
