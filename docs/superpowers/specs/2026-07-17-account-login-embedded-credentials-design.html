account login with embedded credentials & self-installing integration record

Date: 2026-07-17  •  Status: approved design, pending implementation plan

Goal

A user (or agent) on a fresh machine reaches a working, fully-authenticated netsuite-cli against any NetSuite account with a single command and zero NetSuite-side admin setup: 

netsuite-cli account login myaccount

This covers both REST (SuiteQL, records) via OAuth 2.0 and SOAP saved-search via TBA. It works by embedding default integration-record credentials in the binary and relying on NetSuite’s automatic installation of integration records: the first authorization against an unknown client id / consumer key auto-installs the integration record into the target account, where its admins can inspect or block it.

Prior art: ns-metadata, which embeds a public OAuth2 client id and authenticates into arbitrary accounts via browser PKCE flow with account discovery from the callback.

Decision record: embedded credentials

Accepted risk (Jon, 2026-07-17): the TBA consumer key and consumer secret ship in the binary and its public source. This reverses the 2026-07-14 decision against distributing credentials.

  • OAuth 2.0 needs no secret at all: the integration record is a public client using authorization-code + PKCE. Only the client id (non-secret by OAuth design) is embedded.
  • TBA has no public-client equivalent: every TBA request — including the token-issuance handshake — is HMAC-SHA256-signed with consumer_secret&token_secret as the key, so zero-setup TBA requires the consumer secret in the binary.
  • What the published pair grants: nothing by itself. Signing requires a token pair that only exists after a real user authenticates in NetSuite. The practical exposure is that anyone can initiate a consent flow presenting itself as “netsuite-cli”; account admins can block the auto-installed record at any time.
  • Committed plainly, not build-time-injected: the values are extractable from public release binaries regardless, so injection ceremony adds no security. A comment at the definition site records this decision.
Identifier hygiene still applies to everything else. The developer account id where the record lives must never appear in this repo (docs, code, sweep-command examples). The CLI never needs it: auto-install works from the client id / consumer key alone. This document deliberately uses placeholders.

One-time setup: the integration record

One new integration record named netsuite-cli, created in the <developer account>, enabling all three auth methods. Creating it is a manual (or supervised-browser) step for Jon — autonomous creation of credential-issuing records is intentionally out of scope.

Record-save emits the OAuth2 client id and the TBA consumer key/secret exactly once — these three values become the embedded constants.

Embedded constants

// src/auth/embedded.rs (new)
pub const DEFAULT_OAUTH2_CLIENT_ID: &str = "<client id>";
pub const DEFAULT_TBA_CONSUMER_KEY: &str = "<consumer key>";
pub const DEFAULT_TBA_CONSUMER_SECRET: &str = "<consumer secret>";

Resolution order everywhere the CLI needs these values: explicit flag → environment variable (NETSUITE_CLI_TBA_CONSUMER_KEY/SECRET, existing) → stored value (keyring) → embedded default. Accounts that want their own integration record simply keep supplying their own credentials; nothing about existing setups changes.

New command: account login

netsuite-cli account login <alias> [--account-id <id>] [--client-id <id>] [--paste]

(No --port: the callback port is fixed at 8899 because the registered redirect URI must match exactly — same constraint as the existing auth-code flow.)

  1. Auth-code PKCE browser flow with the embedded (or overridden) client id. Without --account-id the authorize URL targets system.netsuite.com and NetSuite lets the user pick the account and role; the callback’s company parameter tells the CLI which account was chosen (account discovery). With --account-id, the login page is pinned to that account.
  2. Store the token + discovered account id in the existing auth-code alias storage (keyring), identical to account add --flow auth-code output.
  3. Offer TBA for saved-search: on a TTY, prompt to chain soap-auth now (mirroring account add’s existing chaining behavior); non-interactive, print the one-line stderr tip. The TBA flow uses the embedded consumer pair unless env/prompt overrides are present; its first request auto-installs the record in the target account.

login is orchestration only: it reuses the existing auth-code machinery (HTTPS loopback on 8899, --paste fallback) and the existing soap-auth implementation. No new auth protocol code.

What does not change

  • account add keeps all current behavior; its --client-id becomes optional for --flow auth-code (falls back to the embedded default) — a consistency win, nothing else.
  • M2M remains per-account setup (certificate mapping is inherently admin work in each target account) and remains the recommended flow for agents: auth-code refresh tokens are one-time rotating, so parallel agent invocations on an auth-code alias still race. login is the human / quick-start path.
  • Existing aliases and their integration records are untouched.

Validation plan

  • Wiremock: new tests for login orchestration — account-id discovery from the callback, credential resolution order (flag > env > embedded), TTY vs non-TTY TBA chaining.
  • Live (the real unknowns are NetSuite-side): from a machine/alias with zero prior setup, run account login against (a) CP sandbox and (b) production; then suiteql and saved-search run; verify the integration record auto-installed into the target account’s Integration list for both OAuth2 and TBA.
  • Horizon check: TBA sunset — no new TBA integrations after NetSuite 2027.1 and SOAP removal in 2028.2. Confirm before 2027.1 whether auto-install into previously-untouched accounts keeps working; saved-search-over-SOAP already carries this caveat in the skill.
