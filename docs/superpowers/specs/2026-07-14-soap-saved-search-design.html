netsuite-cli Saved-Search Execution via SOAP + TBA — Design Spec

2026-07-14 · Adds saved-search run backed by SuiteTalk SOAP, with TBA enabled on the org's existing M2M integration record and browser-based token minting · Goal: agents execute NetSuite saved searches from the CLI, closing the main gap that keeps the MCP connector around.

1. Motivation & decision history

NetSuite's REST APIs cannot execute saved searches; the legacy SOAP (SuiteTalk) API can. Today the CP NetSuite MCP connector is kept alive solely for ns_runSavedSearch and report execution. This feature gives the CLI native saved-search execution over SOAP.

DecisionChoiceRationale
TransportSOAP search/searchMoreWithId with savedSearchId, not a deployed RESTletZero server-side footprint beyond the integration record the CLI already requires. RESTlet alternative rejected for now (needs a SuiteScript deployment per account). Accepted trade-off: SOAP endpoints are removed in NetSuite 2028.2 — this is a ~2-year bridge.
AuthTBA (OAuth 1.0a) via the three-step browser authorization flowSOAP does not accept OAuth 2.0 in any flow. TBA tokens never expire, so it is one browser consent per account, then headless forever. Existing OAuth 2.0 M2M REST auth is untouched.
Integration recordReuse the org's existing M2M record — enable the TBA checkboxes on it; nothing CP-owned is distributed or embeddedThe CLI already requires each org to set up an integration record for M2M OAuth 2.0, so TBA rides the same record and the same setup ritual. No consumer key or secret ships in binaries or the repo; no auto-installed CP record in strangers' accounts. Rejected alternatives: embedded key+secret (public secret enables completing signatures for any leaked token pair, and silent issuetoken minting while that endpoint lives), embedded key + pasted CP secret (still requires CP to distribute a secret; pointless once every org has its own record anyway).
ScopeRun onlyNo SOAP getSavedSearch listing, no ad-hoc criteria overlay. Discovery stays on existing paths. YAGNI.
SOAP clientHand-rolled envelopes + quick-xml parsingOnly 2 of SOAP's 84 operations are used; WSDL codegen and Rust SOAP crates rejected (huge generated surface / unmaintained).

2. Deprecation timeline (accepted)

  • 2027.1 — no new integrations may be created using TBA or SOAP. Each org's integration record must have TBA enabled before then; existing records keep working. (Enable the checkboxes when this ships — don't defer.)
  • 2028.2 — all SOAP endpoints disabled; this feature stops working. The README and skill must state this sunset.
  • Sources: SOAP Removal Plans FAQ, Preparing for TBA End of Support, Auto-Installation of Integration Records.

3. Command surface

CommandBehaviorOutput
saved-search run <id> --type <recordtype> [--limit N] [--all]SOAP search with xsi:type from the record-type table. Numeric <id> → savedSearchId attribute; customsearch_* → savedSearchScriptId. --limit = SOAP page size (5–1000, default 1000). --all follows searchMoreWithId to the last page, with the same runaway-page cap pattern as suiteql --all.{"items":[…], "totalRecords":n, "totalPages":n, "pageIndex":n, "hasMore":bool}; with --all, merged items and "hasMore":false.
account soap-auth <alias>Explicitly runs the three-step TBA browser flow for an already-registered account alias and stores the token. Used for re-auth (e.g. after sandbox refresh) or pre-provisioning.Confirmation JSON (account, entity, role, token stored).
  • First-run auto-auth: saved-search run with no stored TBA token for the account triggers the browser flow automatically when stdin/stderr is a TTY; in non-interactive contexts it exits with the auth-error code and a message naming account soap-auth. If the alias has no consumer key/secret stored yet, the interactive flow first prompts for them (secret with hidden input) and stores them before opening the browser.
  • --type is required: NetSuite has no API to resolve a saved search's record type from its id. Values are lowercase names of the 95 *SearchAdvanced types (e.g. transaction, customer, item, supportcase, employee). Unknown value → usage error listing valid types.
  • The saved search executes with its own saved filters and columns (returnSearchColumns=true); the identity is the user+role that consented, so results honor that role's permissions.

4. Auth design

4.1 Credential matrix

CredentialScopeSourceStorage
TBA consumer keyPer account alias (aliases of the same org share the same value — it is the record's client id)Prompted on first SOAP use / account soap-auth; NETSUITE_CLI_TBA_CONSUMER_KEY env var overrides for CIOS keyring, per alias (same place the existing flows keep client_id — config.toml holds only account_id + flow)
TBA consumer secretPer account alias (same sharing note)Prompted with hidden input alongside the key; NETSUITE_CLI_TBA_CONSUMER_SECRET env var overrides for CI. Never embedded, never committed.OS keyring, per alias
TBA token id + secretPer account alias (bound to consenting user+role)Three-step browser flowOS keyring, per alias, alongside existing M2M secrets

Distribution model: identical to M2M — every org (CP included) uses its own integration record and never receives credentials from anyone else. Enabling TBA on the record is a one-time checkbox edit folded into the existing README setup instructions. Within an org, the consent flow still auto-installs the record into sibling accounts (e.g. sandboxes) that don't have it yet.

OAuth 2.0 M2M for REST is untouched: same integration record may carry both auth methods; M2M certificate mappings remain a manual per-account admin step by NetSuite design (only cert rotation has an API). REST and SOAP may run as different identities.

4.2 Three-step TBA flow (all signatures HMAC-SHA256 per RFC 5849)

  1. Request token: POST https://<acct>.restlets.api.netsuite.com/rest/requesttoken with OAuth 1.0a header (oauth_consumer_key, oauth_callback=http://localhost:<port>, nonce ≥6 chars — we use 20, timestamp, signature keyed by urlencode(consumerSecret)&). Response: request token + secret + oauth_callback_confirmed=true.
  2. Browser consent: open https://<acct>.app.netsuite.com/app/login/secure/authorizetoken.nl?oauth_token=…&state=…. User logs in / picks role / clicks Allow — this consent auto-installs the integration record into the account (State per SOAP Web Services Preferences "Require Approval"). NetSuite redirects to the local callback with oauth_token, oauth_verifier, company, role, entity, state (validate state).
  3. Access token: POST https://<acct>.restlets.api.netsuite.com/rest/accesstoken with oauth_verifier, signature keyed by consumerSecret&requestTokenSecret. Response: permanent oauth_token + oauth_token_secret → keyring.
  • Reuses the local loopback HTTP listener pattern from the existing authcode flow (src/auth/authcode.rs).
  • The integration record's TBA callback URL must exactly match the oauth_callback the CLI sends — a fixed localhost port, registered on the record (setup checklist below).
  • Account-id forms: URLs use the lowercase hyphenated id (1234567-sb1.restlets.api.netsuite.com); the SOAP tokenPassport<account> and OAuth realm use the uppercase underscore form (1234567_SB1). Derived from the alias's configured account id.

4.3 One-time integration-record setup (manual, documented in README)

  • On the org's existing M2M integration record: check Token-Based Authentication and TBA: Authorization Flow; set the TBA callback URL to the CLI's localhost callback; capture the consumer key/secret shown on save and keep them in the org's password manager. Consumer key = the record's client id already in CLI config.
  • Reset Credentials trap: the consumer secret was displayed exactly once when the record was created. If it wasn't captured then, the only ways forward are (a) Reset Credentials — which rotates the client id, breaking every M2M certificate mapping and configured client_id, all of which must then be redone — or (b) a separate integration record just for TBA/SOAP. The README must call this out so orgs choose deliberately.

5. SOAP engine

  • Endpoint: POST https://<acct>.suitetalk.api.netsuite.com/services/NetSuitePort_2025_2, headers Content-Type: text/xml; charset=utf-8 and SOAPAction: "search" / "searchMoreWithId". WSDL version pinned to 2025_2 in namespace URNs.
  • SOAP header — tokenPassport (rebuilt per request): account, consumerKey, token, 20-char alphanumeric nonce, epoch-seconds timestamp, and <signature algorithm="HMAC-SHA256"> = Base64(HMAC-SHA256(key=consumerSecret&tokenSecret, msg=account&consumerKey&token&nonce&timestamp)).
  • SOAP header — searchPreferences: pageSize from --limit, returnSearchColumns=true.
  • Body: 
    <search xmlns="urn:messages_2025_2.platform.webservices.netsuite.com">
  <searchRecord xmlns:q1="urn:sales_2025_2.transactions.webservices.netsuite.com"
    xsi:type="q1:TransactionSearchAdvanced" savedSearchId="57"/>
</search>
    (or savedSearchScriptId="customsearch_x"). searchMoreWithId takes the prior response's searchId + pageIndex.
  • Record-type table: static Rust table mapping the 95 lowercase type names → (xsi:type name, namespace URN), generated from the 2025.2 XSDs (each *SearchAdvanced lives in the module XSD that defines it; the generation script/notes live next to the table for the next WSDL bump).
  • Row flattening (generic, no per-type response types): each searchRow child is either basic or a join (customerJoin, …). For each field element: key = local element name (joins prefixed customer.email — join element name minus the Join suffix); value = searchValue text, else internalId attribute on searchValue (RecordRefs); repeated searchValue children → JSON array; customFieldList/customField keyed by its scriptId. Values stay strings (NetSuite returns typed text; no client-side coercion).

6. Errors

  • SOAP faults (soapenv:Fault, incl. InvalidCredentialsFault/InsufficientPermissionFault) and unsuccessful <status isSuccess="false"> responses are parsed into the existing error enum: auth-class failures use the same exit code as REST auth failures; NetSuite-reported errors (bad saved-search id, no permission on the search) map like REST API errors, with NetSuite's statusDetail code/message passed through; unknown type / bad flags are usage errors (exit 2). No new exit codes.
  • Missing consumer key/secret or TBA token in a non-interactive context → auth-error exit naming account soap-auth and the env-var overrides.

8. Testing

  • Unit: tokenPassport signature against a fixed vector; OAuth 1.0a header signing (known key/nonce/timestamp → known signature); row flattening (basic fields, RecordRef internalId, multiselect arrays, joins, custom fields); record-type lookup.
  • Integration (wiremock, canned XML): happy path single page; --all multi-page via searchMoreWithId; isSuccess=false with statusDetail; SOAP fault; script-id vs internal-id attribute selection; non-TTY missing-token error.
  • Live smoke (gated like live_smoke.rs): run a known sandbox saved search end-to-end; include one custom-record saved search (CustomRecordSearchAdvanced with savedSearchId alone is confirmed working in the account).

  • README: saved-search section — extend the existing integration-record setup docs with the TBA checkboxes + callback URL, the Reset Credentials trap, consumer key/secret prompt + per-account browser consent, sandbox-refresh re-auth caveat, 2028.2 sunset warning.

10. Out of scope

  • SOAP saved-search listing (getSavedSearch) and ad-hoc criteria overlay.
  • TBA as a REST auth fallback (deprecated path; M2M remains the REST auth).
  • Report execution (no SOAP/REST API exists; MCP connector keeps that job).
  • Any RESTlet-based fallback for the 2028.2 sunset — revisit nearer the date.
