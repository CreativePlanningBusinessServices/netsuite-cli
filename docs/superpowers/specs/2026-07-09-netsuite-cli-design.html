1. Purpose & goals

A single-binary CLI, netsuite-cli, that gives AI agents full programmatic access to the NetSuite REST API: record CRUD, SuiteQL, the metadata catalog, RESTlet execution, async job handling, and a raw-request escape hatch. Each invocation targets one of several remembered accounts; credentials are stored securely per account. Output is machine-readable JSON with deterministic exit codes.

A core requirement is surviving NetSuite's twice-yearly API releases without code changes: the CLI is dynamic — record types are runtime arguments and schemas are discovered from each account's own metadata catalog, never hardcoded or code-generated.

3. Command surface

# Accounts
netsuite-cli account add <alias> --account-id 1234567 --flow m2m|auth-code [flags]
netsuite-cli account list | remove <alias> | set-default <alias> | test [<alias>]

# Records (record type is a runtime string, e.g. customer, salesOrder)
netsuite-cli record get    <type> <id> [--fields a,b] [--expand-sub-resources]
netsuite-cli record list   <type> [--q "<filter>"] [--limit N] [--offset N] [--all]
netsuite-cli record create <type> --data '<json>' | --data @file.json | --data -
netsuite-cli record update <type> <id> --data ...
netsuite-cli record upsert <type> <external-id> --data ...
netsuite-cli record delete <type> <id>

# SuiteQL (auto-pagination with --all)
netsuite-cli suiteql "SELECT id, entityid FROM customer" [--limit N] [--offset N] [--all]

# Metadata discovery (cached locally, --refresh to bust)
netsuite-cli describe --list            # all record types in this account
netsuite-cli describe <type>            # field/sublist schema for one type

# RESTlets
netsuite-cli restlet call --script <id> --deploy <id> [--method GET|POST|PUT|DELETE] [--data ...]

# Async jobs
netsuite-cli job submit <METHOD> <path> [--data ...]    # Prefer: respond-async
netsuite-cli job status <job-id>
netsuite-cli job result <job-id>

# Escape hatch — any REST path with account auth applied
netsuite-cli raw GET|POST|PATCH|PUT|DELETE <path> [--data ...] [--header k:v]

# Maintenance
netsuite-cli update [--check]           # self-update from GitHub Releases
netsuite-cli config get|set <key> [value]

Account targeting precedence: --account <alias> flag → NETSUITE_ACCOUNT env var → configured default. Every command behaves identically regardless of which account is selected.

4. Auth & credential storage

M2M (client credentials) — primary, unattended

  • Setup: user creates an Integration record in NetSuite, generates/uploads a certificate, then runs account add --flow m2m supplying client ID, certificate ID, and private-key path. The key is read once and moved into the keychain.
  • Runtime: CLI signs a JWT assertion with the stored private key and exchanges it at the account's OAuth2 token endpoint (.../services/rest/auth/oauth2/v1/token) for an access token (~60 min).

Auth-code — interactive, named user

  • Setup: account add --flow auth-code starts a localhost redirect listener, opens the browser to NetSuite's consent page, and stores the resulting refresh token in the keychain.
  • Runtime: access tokens minted from the refresh token; if the refresh token itself expires, the CLI fails with a structured error telling the agent/user to re-run account test --reauth.

Storage split

  • Keychain (service netsuite-cli, one entry per account alias): private keys, client secrets, refresh tokens, cached access tokens.
  • Config file (TOML in the platform config dir via directories): account aliases, NetSuite account IDs, flow type, default account, cache settings. No secrets, ever.
  • Access tokens auto-refresh transparently; exactly one retry after re-auth on a 401.

5. Agent ergonomics

  • stdout: JSON, always (compact by default, --pretty for humans). Response bodies pass through unmodified; list commands with --all merge pages into one items array.
  • stderr: structured JSON errors: { "kind": "api|auth|network|usage", "status": 400, "message": "...", "details": [...] } with NetSuite's o:errorDetails mapped through.
  • Exit codes: 0 success · 1 API error · 2 usage error · 3 auth error · 4 network error.
  • Discoverability: every subcommand's --help includes copy-pasteable examples; describe gives live per-account schemas so agents never guess field names.

6. Internals

src/
  main.rs          # clap entry, exit-code mapping
  cli.rs           # clap derive command tree
  commands/        # one module per subcommand group (thin: parse → client call → print)
  client.rs        # NsClient: reqwest+tokio, auth middleware, retry/backoff, pagination
  auth/            # m2m JWT assertion, auth-code flow, token cache
  store/           # keyring secrets + TOML config + metadata cache (platform cache dir)
  output.rs        # JSON printing, error envelope, exit codes
  • Key crates: clap (derive), tokio, reqwest (rustls), serde/serde_json, keyring, jsonwebtoken, directories, self_update, thiserror.
  • Retries: 429 and 5xx with exponential backoff, honoring Retry-After; bounded attempts.
  • Metadata cache: per account + record type in the platform cache dir with TTL; --refresh forces a refetch.

7. Testing & CI

  • TDD throughout (superpowers test-driven-development).
  • Unit tests: JWT assertion construction, config/keychain abstraction (trait-mocked), error mapping, exit codes.
  • Integration tests: wiremock fake NetSuite covering auth exchange, CRUD, SuiteQL pagination, 429 retry, async job lifecycle.
  • Optional live smoke tests against a real sandbox, gated behind NETSUITE_LIVE_TEST=1.
  • CI (GitHub Actions): build + test on macOS and Windows; on tag, publish release binaries for both platforms (targets for self_update).

To verify during planning (agent-browser triage of Oracle docs)

  • Exact JWT assertion claims/algorithms (PS256 vs ES256) and token endpoint URL shape for M2M.
  • Metadata catalog paths and required Accept headers (JSON schema vs OpenAPI variants).
  • Async endpoints: Prefer: respond-async support and job-status/result URL shapes.
  • RESTlet base URL (*.restlets.api.netsuite.com) and OAuth2 scope requirements (rest_webservices, restlets, suite_analytics).
  • SuiteQL pagination envelope (links, hasMore, offset) details.

8. Out of scope (v1)

  • OAuth 1.0a / Token-Based Authentication.
  • Code generation from the REST API Browser.
  • SOAP/SuiteTalk, analytics workbooks, file cabinet uploads (reachable via raw if needed).
  • Linux builds (nothing precludes them; just not a target).
