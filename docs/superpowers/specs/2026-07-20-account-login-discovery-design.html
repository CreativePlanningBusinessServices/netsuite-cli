Goal

One command takes a user from a fresh machine to an authenticated CLI against any NetSuite account, without knowing the account id in advance: 

netsuite-cli account login myaccount

The browser opens, the user logs in and picks an account/role on NetSuite’s own chooser, and the CLI learns which account was chosen from the OAuth callback. No --account-id, no --client-id, no NetSuite UI setup (the built-in client id from PR #10 auto-installs its integration record into the target account on first consent).

Scope history

  • 2026-07-17: original design bundled this command with embedded TBA consumer credentials (GitHub-secret injection, binary obfuscation) for zero-setup saved-search.
  • 2026-07-20 (Jon): PR #10 (merged) shipped the built-in OAuth2 client id and UI-free M2M bootstrap via the certificate rotation API, and made the case for keeping TBA off the built-in record (a TBA credential reset rotates the client id, which would invalidate every distributed binary’s embedded id and every M2M cert mapping). Jon adopted that position: no TBA consumer secret is embedded. Saved-search keeps its bring-your-own-credentials TBA setup on a separate record. With no secret in the binary, the GitHub-secret/obfuscation machinery from the original design is dropped, and the original design’s claim that M2M can never be zero-touch is corrected — PR #10’s account cert commands already do exactly that. What remains is this spec: account login with account discovery.

Prerequisites (already in place)

  • PR #10 merged: src/builtin.rs (built-in client id + resolution order), entity/role capture from the OAuth callback, account cert bootstrap.
  • NETSUITE_CLI_BUILTIN_CLIENT_ID repository variable is set, so release builds embed the org’s public-client id.
  • rest_base already normalizes NetSuite account ids of the 1234567_SB1 form into URL hosts.

Command

netsuite-cli account login <alias> [--account-id <id>] [--client-id <id>] [--paste]
  1. Resolve the client id with PR #10’s existing order: explicit --client-id → id already stored for this alias (re-login keeps the integration the alias was registered against) → built-in id. No client id anywhere → the same actionable usage error account add gives.
  2. Browser PKCE flow with account discovery. Without --account-id, the authorize URL targets https://system.netsuite.com/app/login/oauth2/authorize.nl and NetSuite presents its account/role chooser during login; the callback’s company parameter identifies the chosen account. With --account-id, the authorize URL is pinned to that account’s host as today. The token-endpoint URL is therefore built after the callback, from the discovered (or pinned) account id.
  3. Store exactly as account add --flow auth-code does today (config entry with captured entity/role ids, token in the keyring), so the alias immediately works with every other command and with the account cert M2M bootstrap.
  4. Offer TBA chaining identically to account add’s existing behavior (TTY prompt / non-TTY stderr tip) — with the user’s own TBA consumer pair, never a built-in one. Nothing new here; it is the same soap-auth path.

No --port: the callback port stays fixed at 8899 because the registered redirect URI must match exactly.

Implementation shape

PieceChange
src/auth/authcode.rs AuthCodeGrant/LoginOutcome gain company: Option<String> parsed from the callback (alongside the existing entity/role capture). The login flow is restructured so the token URL can be derived post-callback; the existing pre-pinned path is the degenerate case where the account id is known up front.
src/cli.rs New AccountAction::Login variant + dispatch. Reuses resolve_client_id_for_alias unchanged.
src/commands/account.rs New login() orchestration: discovery-capable flow → normalize the callback’s company value (1234567_SB2 form) to the stored account-id convention → store_auth_code_account → existing TBA chain offer. No new auth protocol code.
Docs/skill README quick-start gains the one-command path; SKILL.md bootstrap section leads with account login; account add examples unchanged.

What does not change

  • account add (both flows), account cert, and all existing aliases behave exactly as on current main.
  • M2M remains the recommended flow for agents (auth-code refresh tokens rotate one-time; parallel invocations race). login is the human quick-start and the first step of the M2M bootstrap.
  • Saved-search TBA setup still requires the separate TBA record’s consumer pair.

Validation plan

  • Unit/wiremock: callback parsing with/without company; account-id normalization cases (production, _SB1, _RP forms); login orchestration storing the discovered id; client-id resolution reuse; usage error when no client id is available.
  • Live: account login with no flags against (a) SB2 and (b) an account with no prior netsuite-cli integration record, verifying the record auto-installs and the discovered account id round-trips into working suiteql calls.
