First-Run TBA Onboarding — Design Spec

2026-07-15 · New users should configure ONE integration record for both M2M OAuth 2.0 and TBA at creation time, and learn during account add that saved searches need a separate SOAP token — instead of discovering it weeks later after the one-time consumer secret is gone.

1. Problem

Today account add collects only OAuth 2.0 credentials; TBA surfaces lazily at the first saved-search run. The README's M2M checklist says "Note the Client ID" but never mentions enabling the TBA checkboxes or capturing the Client Secret — which NetSuite displays exactly once at record creation. The default new-user path therefore creates a TBA-less record, discards the secret, and later faces either a record edit + missing secret or the documented Reset Credentials trap (client-id rotation breaking M2M). Chosen approach (Jon, 2026-07-15): docs unification + post-add stderr tip + interactive TTY offer ("Approach B").

2. README restructure — one-save record creation

  • The "NetSuite setup" intro's integration-record paragraph becomes a unified checklist executed in a single save: enable OAuth 2.0 Client Credentials Grant + REST Web Services/RESTlets scopes (and/or Authorization Code Grant for that flow), Token-Based Authentication, TBA: Authorization Flow, and TBA Callback URL https://localhost:8899/callback.
  • Immediately after "save": copy both the Client ID and the Client Secret, with a bold note that the secret is displayed only this once, is exactly what saved-search (SOAP/TBA) auth asks for later, and belongs in a password manager.
  • The M2M and auth-code sections drop their own record-creation steps and reference the checklist; each keeps only its flow-specific steps (certificate/mapping; public client + redirect URI).
  • The "Saved searches (SOAP)" section reframes its setup as "already complete if you followed the setup checklist", keeps the sunset warning, and keeps the Reset Credentials trap as the recovery path for pre-existing records created without TBA.

3. CLI: account add offers SOAP setup

Condition after a successful account add (both flows)Behavior
Alias already has a minted TBA token (get_tba → token_id present)Nothing — no prompt, no tip (re-adds shouldn't nag).
stdin is a TTY, no tokenstderr prompt: Saved searches use SOAP/TBA auth. Set it up for '<alias>' now? [y/N] . On y/Y/yes: run the existing account::soap_auth inline (consumer key/secret prompt → browser consent). Anything else: print the tip.
Not a TTY, no tokenstderr tip only: tip: saved searches authenticate separately (SOAP/TBA) — run `netsuite-cli account soap-auth <alias>` when ready (needs the consumer key/secret captured at integration-record creation). Agents/scripts are never prompted.
  • The offer lives in the cli.rs dispatch (a small helper called from both AccountAction::Add arms after success), reusing commands::account::soap_auth with the add's port/paste arguments where applicable (default 8899, non-paste).
  • The keyring read for the skip-check reuses the store handle already present in the dispatch.

4. Output contract

  • stdout remains exactly one JSON object: the add result, gaining "soapTokenStored": true|false only when the chained setup actually ran (declined/skip paths leave the add JSON unchanged).
  • Chained soap-auth failure does not fail the add: exit 0, "soapTokenStored": false, and the soap error JSON plus re-run: netsuite-cli account soap-auth <alias> guidance on stderr. The primary operation succeeded; the retry path is explicit.
  • No new error kinds; exit codes unchanged.

5. Tests, docs, constraints

  • Pure helper (e.g. soap_auth_tip(alias) -> String and the yes/no answer parser) with unit tests; the TTY prompt branch itself stays untested, matching the sanctioned ensure_tba_secrets precedent.
  • A test covering the merge: add-result JSON + soapTokenStored flag composition.
  • SKILL.md: one line telling agents account add never prompts non-interactively (tip on stderr only) and that account soap-auth remains the explicit path.
  • House rules apply: gate (fmt/clippy -D warnings/test) before every commit, no Creative Planning identifiers in any artifact, copy-paste-correct after_help/README examples.

6. Out of scope

  • Auto-detecting whether the integration record actually has TBA enabled (no API for it).
  • Any change to saved-search run auto-auth, soap_auth internals, or credential storage.
  • Windows/`--paste` variations beyond what soap_auth already handles.
