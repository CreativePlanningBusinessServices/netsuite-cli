netsuite-cli Special Operations — Design Spec

2026-07-14 · Adds first-class subcommands for NetSuite REST "special" operations currently reachable only via raw · Goal: agents discover these capabilities through --help and the skill, and never hand-build the custom Oracle mediaType headers.
Spec review: Approve — write the implementation plan Request changes

1. Motivation & scope

Five NetSuite REST operations from the SuiteTalk help (create-form/edit-form, selectOptions, attach/detach, serverTime, governanceLimits) plus transform execution are all reachable today through raw, but nothing tells an agent they exist, and three of them require the fiddly Accept: application/vnd.oracle.resource+json; type=… mediaType that agents get wrong. This change encodes each as a first-class subcommand (Approach A — full coverage, namespace-aligned) and documents them in README + SKILL.md.

  • External ids work for free: all id positionals pass through verbatim, so NetSuite's eid:ACME-001 form works in every position (documented, not special-cased).
  • Attach direction: the first pair is the record being attached to (e.g. record attach customer 660 contact 106 attaches contact 106 to customer 660) — after_help examples make this unambiguous.
  • Only attach supports --role (per the doc page); detach takes no options.
  • On record transform, --fields/--expand-sub-resources shape the form response and are only valid together with --form; using them without it is a usage error (exit 2).

3. Behavior details

  • Accept headers set internally. Commands pass the mediaType through the existing NsClient::request headers slice, same as describe.rs does with Accept: application/json. No client changes needed (the reqwest client sets no default Accept).
  • --data optional everywhere here, accepting inline JSON / @file / - via the existing read_data_arg. When omitted on POST/PATCH-based ops, send {} — matches Oracle's own examples and avoids empty-body edge cases.
  • --q passes through raw (e.g. entitystatus START_WITH LEAD-, location IS "United States") exactly like record list --q; no client-side parsing.
  • Errors: no new error kinds; API failures flow through the existing api_error mapping and exit-code contract (1/2/3/4).
  • Mutation safety: transform without --form, attach, and detach mutate; help text says so plainly (matching the "no surprising side effects" stance of restlet call --method).

5. Testing

  • Wiremock integration tests: new tests/record_special_ops.rs and tests/system_commands.rs, following the existing suites — assert method, full path (incl. !transform/!attach/!detach segments), Accept header value, query params, request body (incl. the {} default and --role shape), and output JSON (incl. 204 handling, Location-derived id, and the create-like error when Location is absent).
  • CLI parse tests in cli.rs for the new verbs (e.g. select-options requires --fields; attach arg order).
  • Live smoke: add read-only ops only — system server-time, system governance-limits, record select-options on a safe type. Mutating ops (transform/attach/detach) stay wiremock-only.
  • Gate: cargo fmt --check · cargo clippy --all-targets -- -D warnings · cargo test.

6. Documentation (the discoverability goal)

  • README: extend the record section with transform / forms / select-options / attach-detach examples; new system section; keep the existing structure and tone.
  • SKILL.md: new "Pick the right command" rows — "Turn one record into another (SO→invoice)" → record transform; "Preview defaults before create/update" → record create-form/edit-form; "Valid dropdown values for a field" → record select-options; "Link/unlink a contact or file" → record attach/detach; "Server clock / concurrency limits" → system …. Short recipe note: --form previews a transform without creating; select-options --data carries dependent-field context. The existing batch-collections recipe stays on raw, unchanged.

7. Constraints & non-goals

  • No changes to NsClient, auth, error kinds, or the output contract.
  • No Creative Planning account identifiers in any repo artifact (public repo).
  • Non-goals: no batch/collection convenience verbs (stays a raw recipe), no saved-search or report support (NetSuite has no REST API for those), no --expand pass-through beyond the flags listed (add later if needed).
  • Delivery: feature branch → PR → pr-review-loop to green.
